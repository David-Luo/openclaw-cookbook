## 一、问题定义

### 1.1 背景与动机

**用户痛点**：

| 痛点 | 描述 | 影响 |
|-----|------|------|
| 命题模糊 | 研究主题不够清晰，难以拆解 | 研究方向不确定，浪费时间 |
| 信息分散 | 资料来源多、格式杂、质量参差 | 信息整合困难，遗漏关键内容 |
| 推理浅层 | 分析停留在表面，缺乏深度逻辑链 | 结论不够深入，洞察不足 |
| 输出混乱 | 研究结果缺乏结构化呈现 | 难以复用和传播 |

**目标用户**：
- 需要进行深度研究的技术人员、研究人员
- 可能对研究问题域不熟悉，需要外部帮助建立认知
- 期望获得结构化、有洞察的研究成果

### 1.2 核心问题陈述

本Agent需要解决四个核心问题：

| 问题 | 具体描述 |
|-----|---------|
| Q1：命题解构 | 如何将模糊的研究命题转化为清晰、可操作的研究框架？ |
| Q2：信息整合 | 如何系统性收集、评估、整合多源信息？ |
| Q3：深度推理 | 如何进行多步骤深度推理，形成有洞察的结论？ |
| Q4：报告生成 | 如何将研究成果转化为结构化、高质量的研究报告？ |

### 1.3 价值定位

**核心理念**：研究思维的放大器，而非简单的信息检索工具

| 价值维度 | 用户获益 |
|---------|---------|
| 降低门槛 | 不熟悉问题域的用户也能开展系统性研究 |
| 提升质量 | 结论有证据支撑、推理有逻辑链条、报告有结构 |
| 节省时间 | 自动化执行信息收集和推理过程 |
| 增强洞察 | 多视角分析、深度推理形成创新见解 |

## 二、能力定义

### 2.1 四大核心能力

Agent具备四大核心能力，对应研究流程的不同阶段：

| 能力 | 定义 | 交付物 | 阶段 |
|-----|------|--------|------|
| **命题解构** | 将模糊命题转化为结构化研究框架 | 问题树 + 研究边界 + 初始假设 | 阶段一 |
| **自动化调研** | 系统性收集和评估信息，构建证据链 | 证据链 + 信息质量报告 | 阶段二三 |
| **深度推理** | 多步骤推理分析，形成结论和洞察 | 推理链 + 核心结论 | 阶段二三 |
| **报告生成** | 生成结构化研究报告，包含质检机制 | 研究报告 + 质检报告 | 阶段四 |

### 2.2 命题解构能力详解

**能力描述**：帮助用户将模糊的研究命题转化为清晰、可操作的研究框架

**关键子能力**：
- 命题理解：提取关键词、识别领域、判断意图
- 快速信息收集：轻量级收集领域概况，帮助用户建立基本认知
- 问题域识别：识别边界、提供研究方向建议
- 概念建模：建立概念体系、构建概念关联图谱
- 命题拆解：输出问题树、研究边界、初始假设

**交付物规格**：
```
研究框架 = {
    问题树: 主问题 → 子问题 → 具体问题（MECE结构）
    研究边界: 范围内 + 范围外（明确界定）
    初始假设: H1, H2, H3...（待验证）
}
```

**用户交互**：研究框架需经用户确认后才进入后续阶段

### 2.3 自动化调研能力详解

**能力描述**：系统性收集、评估、整合多源信息，构建结构化证据链

**关键子能力**：
- 信息收集：多源检索、资料抓取、整理归档
- 知识抽取：从原始信息中提取关键事实
- 质量评估：评估信息来源可靠性、内容可信度
- 证据链构建：将零散信息组织为证据→结论的支撑结构

**交付物规格**：
```
证据链 = {
    关键发现: [
        {发现内容, 证据来源, 可信度评级}
    ]
    信息矩阵: 问题 × 信息来源
    质量报告: 权威来源占比、信息覆盖度
}
```

**质量标准**：
- 关键信息至少两个来源验证
- 来源可信度分级（高/中/低）
- 信息冲突需标注并分析原因

### 2.4 深度推理能力详解

**能力描述**：进行多步骤深度推理，从证据到结论，形成有洞察的见解

**关键子能力**：
- 推理策略选择：演绎/归纳/溯因/类比（根据场景选择）
- 多轮迭代推理：假设→验证→修正→再验证
- 模式识别：从散点信息发现隐藏规律
- 认知评估：评估认知完整性，决定是否继续迭代

**交付物规格**：
```
推理产物 = {
    推理路径: [步骤1 → 步骤2 → ... → 结论]
    模式发现: 发现的规律和共性
    核心结论: 可验证的结论（含置信度）
    认知缺口: 待补充的方向
}
```

**推理质量标准**：
- 推理链条可追溯（每步有前提→逻辑→结论）
- 关键结论标注置信度（高/中/低）
- 存在不确定性时明确标注

### 2.5 报告生成能力详解

**能力描述**：将研究成果转化为结构化研究报告，包含质检和修复机制

**关键子能力**：
- 结构设计：根据研究成果设计报告章节
- 内容撰写：整合推理结论和证据
- 图表生成：知识图谱、对比表格、趋势图
- 引用整理：参考文献、信息来源标注
- 质检检查：证据质量、推理逻辑、结论可靠性、报告格式

**交付物规格**：
```
研究报告 = {
    执行摘要: 核心发现、关键结论、建议行动
    正文章节: [概述、现状、挑战、趋势...]
    图表附件: 概念图谱、证据链、对比表
    参考文献: 来源列表（含可信度标注）
}
```

**质检标准**（详见质量约束章节）

## 三、输入输出规格

### 3.1 输入规格

| 输入类型 | 格式 | 必需性 | 说明 |
|---------|------|--------|------|
| 研究命题 | 自然语言文本 | 必需 | 用户提出的待研究主题，如"帮我研究AI测试" |
| 用户反馈 | 选择/文本补充 | 阶段一必需 | 研究框架确认点的用户响应 |
| 补充信息 | 自然语言文本 | 可选 | 用户主动提供的相关背景、约束条件 |

**研究命题示例**：
```
简单命题："调研一下微服务架构"
中等命题："帮我研究AI测试的现状和趋势"
复杂命题："分析云原生技术在金融行业的应用挑战和创新机会"
```

**用户反馈类型**：
| 反馈类型 | 示例 | Agent响应 |
|---------|------|-----------|
| 确认 | "框架没问题，开始研究" | 进入后续阶段 |
| 调整细节 | "Q3问题改成技术挑战" | 增量修改框架 |
| 重置方向 | "研究方向不对，重新拆解" | 完全重新命题解构 |
| 补充信息 | "重点关注大模型测试" | 整合信息更新框架 |

### 3.2 输出规格

| 输出类型 | 格式 | 阶段 | 说明 |
|---------|------|------|------|
| 研究框架 | Markdown结构化文本 | 阶段一 | 问题树 + 边界 + 假设 |
| 最终报告 | Markdown文档 | 阶段四 | 完整研究报告，含图表和参考文献 |
| 中间产物 | Markdown/JSON | 可选 | 概念图谱、证据链（用户可请求） |

**研究框架输出模板**：
```markdown
## 研究问题树
主问题：[核心问题]
├── Q1：[子问题1]
├── Q2：[子问题2]
├── Q3：[子问题3]
└── Q4：[子问题4]

## 研究边界
✓ 范围内：[明确的范围]
✗ 范围外：[排除的范围]

## 初始假设
- H1：[假设1]
- H2：[假设2]
```

**研究报告输出模板**：
```markdown
# [研究主题]研究报告

## 执行摘要
- 核心发现（3条）
- 关键结论（2条）
- 建议行动（3条）

## 一、概述
## 二、现状分析
## 三、挑战与痛点
## 四、趋势与展望
## 五、结论与建议

## 附录
- 参考文献
- 证据链摘要
- 方法论说明
```

### 3.3 交互规格

**交互设计原则**：
- 阶段一：人机协同，用户参与确认
- 阶段二三：自动执行，无需用户干预
- 阶段四：自动执行+质检回流

**唯一交互点**：

| 交互点 | 触发时机 | Agent展示内容 | 用户可选行为 |
|-------|---------|---------------|-------------|
| 研究框架确认 | 阶段一完成 | 问题树 + 边界 + 假设 + 方向建议 | [1]确认开始 [2]调整细节 [3]重置方向 [4]补充信息 |

**交互流程**：
```
Agent输出研究框架 → 等待用户响应 → 
    IF 用户确认 → 进入阶段二三（自动执行）
    ELIF 用户调整 → 增量修改 → 重新展示框架
    ELIF 用户重置 → 回到命题理解 → 重新解构
    ELIF 用户补充 → 整合信息 → 更新框架 → 重新展示
```

## 四、质量约束

### 4.1 必须满足的质量要求

| 约束类型 | 要求 | 说明 |
|---------|------|------|
| 框架确认 | 研究框架必须经用户明确确认 | 未确认不能进入后续阶段 |
| 证据支撑 | 所有结论必须有证据支撑 | 无证据的结论标注为"推测" |
| 质检通过 | 报告必须通过质检才能输出 | 质检失败则回流修复 |
| 来源标注 | 关键信息必须标注来源 | 来源缺失视为不合格 |

### 4.2 性能约束

| 约束 | 阈值 | 说明 |
|-----|------|------|
| 阶段一迭代上限 | 无硬性限制 | 用户可反复调整直到满意 |
| 阶段二三迭代上限 | 最多100轮 | 防止无限循环，达到强制结束 |
| 质检回流上限 | 无硬性限制 | 直到质检通过才输出 |

**迭代上限说明**：
- 阶段一不设硬性上限，因为研究框架的正确性至关重要
- 阶段二三设100轮上限，平衡质量与效率
- 质检不设上限，确保输出质量

### 4.3 交付质量标准

**研究框架质量标准**：
| 检查项 | 标准 |
|-------|------|
| 问题树完整性 | 主问题→子问题→具体问题，MECE结构 |
| 边界清晰性 | 范围内和范围外明确区分 |
| 假设合理性 | 假设数量≥2，每个假设可验证 |

**证据链质量标准**：
| 检查项 | 标准 |
|-------|------|
| 来源可信度 | 关键发现至少1个高可信度来源 |
| 来源多样性 | 重要结论至少2个来源支撑 |
| 冲突处理 | 信息冲突需标注并给出判断依据 |

**报告质量标准**：
| 检查项 | 标准 |
|-------|------|
| 证据质量 | 来源可靠、充分支撑结论 |
| 推理逻辑 | 链条完整、无断裂跳跃 |
| 结论可靠性 | 有依据、无矛盾、不确定性标注 |
| 报告格式 | 结构清晰、图表完整、引用规范 |

### 4.4 异常情况处理原则

| 异常类型 | 处理原则 |
|---------|---------|
| 信息不足 | 标注认知缺口，继续收集或标注"证据不足" |
| 信息冲突 | 分析冲突原因，给出判断依据或标注"存疑" |
| 推理困难 | 标注推理断裂点，回流修复或标注"推理中断" |
| 资源超限 | 达到迭代上限，输出当前最优报告并标注局限 |

## 五、需求验收标准

### 5.1 功能验收

| 验收项 | 验收标准 |
|-------|---------|
| 命题解构 | 能将模糊命题转化为结构化研究框架，经用户确认 |
| 自动化调研 | 能系统性收集信息，构建证据链，输出质量报告 |
| 深度推理 | 能多步骤推理，形成结论，标注置信度 |
| 报告生成 | 能输出结构化报告，通过质检 |

### 5.2 交互验收

| 验收项 | 验收标准 |
|-------|---------|
| 确认机制 | 阶段一有且仅有研究框架确认交互点 |
| 智能判断 | 能区分方向问题（重置）和细节问题（调整） |
| 用户赋能 | 能帮助不熟悉问题域的用户建立基本认知 |

### 5.3 质量验收

| 验收项 | 验收标准 |
|-------|---------|
| 证据支撑 | 报告中结论有证据标注，来源可靠 |
| 质检机制 | 报告经过四维质检，不合格有回流修复 |
| 不确定性标注 | 存在不确定性的结论明确标注 |
