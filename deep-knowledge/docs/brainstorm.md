## 一、设计背景与目标

### 1.1 问题背景

用户需要一个**深度研究认知Agent**，辅助完成特定命题（如"AI测试"）的深度研究。这是OpenClaw Cookbook项目中的一个子项目，需要创建**全新的独立Agent配置**。

核心痛点：
- 研究命题模糊时缺乏结构化拆解方法
- 信息收集分散、缺乏系统性
- 推理过程不够深度、缺乏逻辑链
- 研究成果缺乏结构化输出

### 1.2 设计目标

构建一个**深度研究Agent**，具备：
- **命题解构能力**：将模糊命题转化为结构化研究框架（用户参与确认）
- **自动化研究能力**：迭代执行信息收集和深度推理，直到认知完整
- **报告生成能力**：结构化研究报告输出，包含质检和回流机制

### 1.3 设计约束

| 约束 | 说明 |
|-----|------|
| Agent定位 | 全新独立Agent（新的SOUL.md），非现有Agent增强 |
| 交互方式 | 阶段一人机协同确认，后续阶段自动执行 |
| 五层作用 | 自适应能力调度（根据命题复杂度自动判断） |
| 运行环境 | OpenClaw原生（Markdown+YAML配置） |
| 交付阶段 | 先设计文档，评审通过后再实现 |

## 二、核心流程设计（迭代调整过程）

### 2.1 流程设计迭代历史

**第一版（已废弃）**：四个确认点，全程用户参与
- 研究框架确认、阶段性发现审阅、结论确认、报告结构确认
- 问题：交互过多，效率低下

**第二版（已废弃）**：确认点精简到阶段一，后续自动执行
- 但阶段一缺乏信息收集帮助用户建立认知
- 缺乏质检回流机制

**第三版（当前版本）**：
- 阶段一：快速信息收集 + 用户确认迭代 + 智能判断回流
- 阶段二三：自动迭代循环（最多100轮）
- 阶段四：报告生成 + 四维质检 + 精准回流

### 2.3 四个阶段详细说明

#### 阶段一：命题接纳与解构（用户参与确认）

**核心目标**：识别问题域、研究方向，建立研究框架

**关键特点**：
- **轻量级信息收集**：阶段一的信息收集是快速概览，目的是帮助用户建立基本认知
- **用户提供信息**：用户可能对问题域不熟悉，需要Agent主动补充信息、提供建议
- **迭代确认机制**：直到用户满意确认研究框架，才能进入后续阶段
- **智能判断回流**：方向问题完全重置，细节问题增量调整

#### 阶段二三：信息收集+深度推理迭代循环（自动执行）

**核心目标**：建立方向正确、体系完整的本质认知

**认知完整标准**：
- **概念清晰性**：关键概念都有清晰定义和关联
- **结论形成**：形成了可验证的核心结论和洞察

**迭代策略优化**：
- 每轮迭代后，Agent评估认知缺口，确定下一轮的重点方向
- 动态调整信息收集策略，聚焦未解决的问题
- 避免重复收集相同信息，追求效率

**轮次限制**：
- 最多100轮，防止无限循环
- 达到100轮时强制结束，进入报告生成

#### 阶段四：报告生成+质检（自动执行+回流机制）

**质检回流机制**：
| 问题类型 | 回流目标 | 处理动作 |
|---------|---------|---------|
| 证据质量问题 | 阶段二信息收集 | 补充证据、更换来源 |
| 推理逻辑问题 | 阶段三深度推理 | 修复推理链、补充步骤 |
| 结论可靠性问题 | 阶段三深度推理 | 重塑结论、加强依据 |
| 报告格式问题 | 不回流 | 直接修复报告 |

## 五、设计评审要点（探索阶段）

### 5.1 需确认的问题

1. **阶段一迭代上限**：用户不满意时是否需要设置迭代上限？
   → 决策：无硬性限制，框架正确性至关重要

2. **认知完整标准细化**：当前标准是否足够具体？是否需要量化指标？
   → 决策：概念清晰性 + 结论形成（后续设计文档量化）

3. **回流次数限制**：质检回流是否需要次数限制？
   → 决策：最多5次，始终不通过时给出最终质检结论

4. **进度汇报机制**：阶段二三长时间执行时，是否需要向用户汇报进度？
   → 决策：自动执行不汇报，后续设计文档可讨论

### 5.2 设计亮点总结

| 特性 | 说明 |
|-----|------|
| 用户参与精简 | 只在阶段一确认，后续自动执行 |
| 智能迭代判断 | 方向问题重置，细节问题调整 |
| 自动认知构建 | 迭代循环直到认知完整（最多100轮） |
| 质检回流机制 | 四维质检+精准回流修复 |
| 用户赋能设计 | 快速信息收集帮助不熟悉用户 |
